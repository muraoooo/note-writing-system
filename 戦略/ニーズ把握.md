# ニーズ把握 — 「書きたいテーマ」を「読まれるテーマ」に変換する

## 大原則

**読者は「あなたが何を書きたいか」に1ミリも興味がない。** 自分の悩み・欲求が解決するかだけを見ている。
書き始める前に「これは誰のどの悩みを解決するか」を1文で書けないなら、まだ書くな。

## ニーズの3層構造

| 層 | 内容 | 例(本リポジトリ筆者の領域) |
|---|---|---|
| 顕在ニーズ | 検索される言葉 | 「ローカルLLM 動かし方」「MIT ライセンス 商用」 |
| 潜在ニーズ | 検索しないが抱えている悩み | 「AI使ってみたいけど何から手を付ければいいか分からない」 |
| 感情ニーズ | 共感されたい / 安心したい / 自慢したい | 「自分は時代に置いていかれてないと確認したい」 |

**伸びる記事は3層すべてに応える。** 顕在だけだと検索流入はあっても読了されず、潜在だけだとそもそも見つけられない。

## ニーズ検証チェック(書く前に)

- [ ] このテーマで月100回以上Google/note内検索されているか?
- [ ] 競合記事(上位5本)の最終更新は1年以上前か? → Yesなら新規参入チャンス
- [ ] 自分が3か月前に欲しかった情報か? → 等身大の読者がいる
- [ ] タイトルに「動詞 + 結果」が含められるか?(例:「30分で動かす」「10秒で削除する」)

## ネタ出しソース(枯れない泉)

1. **自分が直近1週間でググった検索履歴** — 同じ悩みの人が必ず存在する
2. **Twitter/Xで「誰か知ってたら教えて」が付いた投稿** — 解決記事が不足している証拠
3. **noteの公式お題・コンテスト** — 露出ブースト付き
4. **GitHub Trending + Hacker News** — 国内日本語記事になる前に書けば独占できる
5. **自分の過去記事のコメント・スキしたユーザーの他のスキ記事** — 隣接ニーズが見える

## やってはいけない

- 「書きたいから書く」を一次動機にする(日記化する)
- 流行語をタイトルに入れるだけで中身が薄い(直帰率が跳ね上がる)
- 「初心者向け」と書いて中級者向けの内容を書く(失望→低評価)
